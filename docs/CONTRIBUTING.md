
# Contributing

Thanks for your interest in contributing to this project! We welcome contributions of all kinds — bug reports, feature requests, documentation improvements, and code. This document explains how to get started and what we expect from contributors.

## Code of Conduct

Please follow the project's [Code of Conduct](CODE_OF_CONDUCT.md). Be respectful and considerate in all interactions.

## How to contribute

- Report bugs or suggest features by opening an issue.
- Search existing issues before creating a new one to avoid duplicates.
- Provide a clear title and description, include steps to reproduce, expected vs actual behavior, and relevant environment details.

## Development workflow

1. Fork the repository and create a branch for your change: `git checkout -b feat/short-description`.
2. Keep commits small and focused. Use present-tense, imperative commit messages (e.g., "Add input validation").
3. Run tests and linters locally before opening a pull request.
4. Push your branch and open a pull request against the `main` branch.

In your pull request:
- Describe the change and the motivation.
- Reference any related issues (e.g., "Fixes #123").
- Add tests if applicable and update documentation when behavior changes.

## Coding standards

- Follow existing code style and patterns used in the repository.
- Write clear, maintainable code and include comments where necessary.
- Keep dependencies minimal and justified.

## Tests

Add or update tests for any bug fixes or new features. Ensure the test suite passes locally before submitting a pull request.

## Documentation

Update the README, inline docs, or other project documentation to reflect any user-visible changes.

## Security issues

If you discover a security vulnerability, please do not open a public issue. Contact the maintainers privately — see the project `README.md` for contact details.

## Reviewing process

Maintainers will review pull requests and provide feedback. Be responsive to review comments and iterate on your changes. Tests and CI must pass before merging.

## License

By contributing, you agree that your contributions will be licensed under the project's license.

## Questions

If you have questions or need guidance, open an issue or contact the maintainers.
